# Tapmeza — Multi-Tenant QR Ordering SaaS

Tapmeza is a multi-tenant QR ordering SaaS for hospitality venues (hotels, restaurants, beach clubs). A customer scans a QR code at their table, sun lounge, or hotel room; a mobile web page opens showing the correct menu for that spot; they order without installing anything or logging in. Venue staff watch incoming orders on a tablet in real time and advance each order through its lifecycle.

## Tech Stack
- **Framework**: Next.js 15 (App Router), TypeScript (strict mode).
- **Styling**: Tailwind CSS v4 (with touch target optimizations).
- **Backend/Database**: Supabase (Postgres + Auth + Realtime + Storage).
- **Package Manager**: `pnpm`.

## Local Setup & Quickstart

To get a local development server running with the full database stack, follow these steps:

### Prerequisites
1. Ensure you have [Node.js](https://nodejs.org/) installed.
2. Ensure you have [Docker](https://www.docker.com/) running on your machine (needed for local Supabase).
3. Ensure you have `pnpm` installed. If not, run `npm install -g pnpm`.

### Installation & Initialization

1. **Install NPM dependencies**:
   ```bash
   pnpm install
   ```

2. **Initialize Local Supabase Stack**:
   Start the local Supabase container environment. This will spin up PostgreSQL, GoTrue Auth, Realtime, Storage, Inbucket, and Studio:
   ```bash
   pnpm supabase start
   ```

3. **Database Migration & Seed**:
   Once the containers start successfully, apply the initial migrations and seeds:
   ```bash
   pnpm supabase db reset
   ```
   *This command runs the `supabase/migrations/0001_init.sql` schema and loads seed data from `supabase/seed.sql`.*

4. **Environment Variables Config**:
   Copy `.env.example` into `.env.local`:
   ```bash
   cp .env.example .env.local
   ```
   Open `.env.local` and replace the placeholder keys with the local development keys generated by the `pnpm supabase start` command (e.g., the local `anon` key and local `service_role` key).

5. **Run the Next.js Development Server**:
   ```bash
   pnpm dev
   ```
   Open [http://localhost:3000](http://localhost:3000) to view the application surfaces.

## Testing the Customer App Flow Locally

1. After running the migrations and seeding successfully, retrieve a scannable URL structure from the local database:
   ```bash
   pnpm supabase db query "SELECT v.slug, l.qr_token FROM locations l JOIN venues v ON v.id = l.venue_id LIMIT 1;"
   ```
2. In your browser, open the returned QR ordering link:
   ```
   http://localhost:3000/o/{slug}/{qr_token}
   ```
   For example, using the default seeded venue:
   ```
   http://localhost:3000/o/tapmeza-beach-club/670df055-6b5f-4d37-8efd-bb7f385c2c58
   ```
3. Verify that the page displays the correct venue slug and location QR token.

## Test on your phone

Since Tapmeza is a mobile-first QR ordering application, it is highly recommended to test the flow on a real physical mobile device connected to your local network (LAN):

1. **Find your local computer IP address**:
   - On macOS, run `ipconfig getifaddr en0` or inspect network Wi-Fi settings to locate your local IP (e.g., `192.168.1.50`).
2. **Bind the Next.js dev server to your LAN**:
   Start the development server binded to all interfaces:
   ```bash
   pnpm dev --hostname 0.0.0.0
   ```
3. **Configure Local Environment**:
   Update `NEXT_PUBLIC_APP_URL` in your `.env.local` file to use your machine's LAN IP address:
   ```env
   NEXT_PUBLIC_APP_URL=http://192.168.1.50:3000
   ```
4. **Scan and Order**:
   - Ensure your smartphone is connected to the same Wi-Fi network.
   - Navigate to the scannable URL on your phone browser:
     `http://<YOUR_LAN_IP>:3000/o/tapmeza-beach-club/670df055-6b5f-4d37-8efd-bb7f385c2c58`
   - Complete the full cart checkouts and order progress polling on your actual device.

## Key Environment Variables

| Variable | Description | Default |
|:---|:---|:---|
| `NEXT_PUBLIC_APP_URL` | Base URL used to build scannable QR codes. Never hardcode the domain. | `http://localhost:3000` |
| `NEXT_PUBLIC_SUPABASE_URL` | Local API Gateway Endpoint for Supabase. | `http://127.0.0.1:54321` |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Public Supabase Key, safe for client bundles. | *Local anon key* |
| `SUPABASE_SERVICE_ROLE_KEY` | Highly sensitive Server-only key used for locked RLS paths. | *Local service key* |
| `GOOGLE_CLIENT_ID` | OAuth credentials client ID (configured in Auth Dashboard). | *Placeholder* |
| `GOOGLE_CLIENT_SECRET` | OAuth credentials client secret (configured in Auth Dashboard). | *Placeholder* |

## Architectural Boundaries & Code Seams

- **Daypart Filtering**: Daypart availability verification checks are executed inside `src/lib/time.ts` (`isWithinDaypart`) factoring in the venue timezone.
- **Integer Money**: Floating-point money variables are banned. Display formatting and minor-unit translations are structured inside `src/lib/money.ts`.
- **Sensitive Database Operations**: Bypassing client-side exposures is achieved via `src/lib/supabase/service-role.ts`, which is restricted with `import 'server-only'` and throws a runtime error if executed client-side.
- **Settlement Providers (Phase 3 Seam)**: Isolated settlement logic is implemented inside the `src/lib/settlement/` seam. Live payment integrations (e.g. ClickPesa/Selcom/DPO/Stripe) can be added directly by extending the `SettlementProvider` interface in `src/lib/settlement/types.ts` without modifying the core order system.
- **Append-only Logs**: Changing or updating lifecycle logs is prevented by `order_events` database level policies.
